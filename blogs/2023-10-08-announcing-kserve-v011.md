<p><em>Published on October 8, 2023</em></p>
<p>We are excited to announce the release of KServe 0.11. In this release we introduced Large Language Model (LLM) runtimes, made enhancements to the KServe control plane, Python SDK Open Inference Protocol support and dependency management. For ModelMesh we have added features PVC, HPA, payload logging to ensure feature parity with KServe.</p>
<p>Here is a summary of the key changes:</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="-kserve-core-inference-enhancements">🚀 KServe Core Inference Enhancements<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-kserve-core-inference-enhancements" title="Direct link to 🚀 KServe Core Inference Enhancements">​</a></h2>
<ul>
<li class="">
<p><strong>Path-based routing support</strong> which is served as an alternative way to the host based routing, the URL of the <code>InferenceService</code> could look like <code>http://&lt;ingress_domain&gt;/serving/&lt;namespace&gt;/&lt;isvc_name&gt;</code>.
Please refer to the <a class="" href="https://github.com/kserve/kserve/blob/294a10495b6b5cda9c64d3e1573b60aec62aceb9/config/configmap/inferenceservice.yaml#L237" rel="noopener noreferrer" target="_blank">doc</a> for how to enable path based routing.</p>
</li>
<li class="">
<p><strong>Priority field for Serving Runtime</strong> custom resource to handle the case when you have multiple serving runtimes which support the same model formats, see more details from <a class="" href="https://kserve.github.io/archive/0.11/modelserving/servingruntimes/#priority" rel="noopener noreferrer" target="_blank">the serving runtime doc</a>.</p>
</li>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-enhanced-python-sdk-dependency-management">📦 Enhanced Python SDK Dependency Management<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-enhanced-python-sdk-dependency-management" title="Direct link to 📦 Enhanced Python SDK Dependency Management">​</a></h3>
<ul>
<li class="">
<p>KServe has adopted <a class="" href="https://python-poetry.org/docs/" rel="noopener noreferrer" target="_blank">poetry</a> to manage python dependencies. You can now install the KServe SDK with locked dependencies using <code>poetry install</code>.
While <code>pip install</code> still works,  we highly recommend using poetry to ensure predictable dependency management.</p>
</li>
<li class="">
<p>The KServe SDK is also slimmed down by making the cloud storage dependency optional, if you require storage dependency for custom serving runtimes you can still install with <code>pip install kserve[storage]</code>.</p>
</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-kserve-python-runtimes-improvements">🔧 KServe Python Runtimes Improvements<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-kserve-python-runtimes-improvements" title="Direct link to 🔧 KServe Python Runtimes Improvements">​</a></h3>
<ul>
<li class="">
<p>KServe Python Runtimes including <a class="" href="https://kserve.github.io/archive/0.11/modelserving/v1beta1/sklearn/v2/" rel="noopener noreferrer" target="_blank">sklearnserver</a>, <a class="" href="https://kserve.github.io/archive/0.11/modelserving/v1beta1/lightgbm/" rel="noopener noreferrer" target="_blank">lgbserver</a>, <a class="" href="https://kserve.github.io/archive/0.11/modelserving/v1beta1/xgboost/" rel="noopener noreferrer" target="_blank">xgbserver</a>
now support the open inference protocol for both REST and gRPC.</p>
</li>
<li class="">
<p>Logging improvements including adding Uvicorn access logging and a default KServe logger.</p>
</li>
<li class="">
<p><code>Postprocess</code> handler has been aligned with open inference protocol, simplifying the underlying transportation protocol complexities.</p>
</li>
</ul>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-llm-runtimes">🤖 LLM Runtimes<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-llm-runtimes" title="Direct link to 🤖 LLM Runtimes">​</a></h3>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="torchserve-llm-runtime">TorchServe LLM Runtime<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#torchserve-llm-runtime" title="Direct link to TorchServe LLM Runtime">​</a></h4>
<p>KServe now integrates with TorchServe 0.8, offering the support for <a class="" href="https://pytorch.org/serve/large_model_inference.html" rel="noopener noreferrer" target="_blank">LLM models</a> that may not fit onto a single GPU.
Huggingface Accelerate and Deepspeed are available options to split the model into multiple partitions over multiple GPUs. You can see the <a class="" href="https://kserve.github.io/archive/0.11/modelserving/v1beta1/llm/torchserve/accelerate/" rel="noopener noreferrer" target="_blank">detailed example</a> for how to serve the LLM on KServe with TorchServe runtime.</p>
<h4 class="anchor anchorTargetStickyNavbar_Vzrq" id="vllm-runtime">vLLM Runtime<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#vllm-runtime" title="Direct link to vLLM Runtime">​</a></h4>
<p>Serving LLM models can be surprisingly slow even on high end GPUs, <a class="" href="https://github.com/vllm-project/vllm" rel="noopener noreferrer" target="_blank">vLLM</a> is a fast and easy-to-use LLM inference engine. It can achieve 10x-20x higher throughput than Huggingface transformers.
It supports <a class="" href="https://www.anyscale.com/blog/continuous-batching-llm-inference" rel="noopener noreferrer" target="_blank">continuous batching</a> for increased throughput and GPU utilization,
<a class="" href="https://vllm.ai/" rel="noopener noreferrer" target="_blank">paged attention</a> to address the memory bottleneck where in the autoregressive decoding process all the attention key value tensors(KV Cache) are kept in the GPU memory to generate next tokens.</p>
<p>In the <a class="" href="https://kserve.github.io/archive/0.11/modelserving/v1beta1/llm/vllm/" rel="noopener noreferrer" target="_blank">example</a> we show how to deploy vLLM on KServe and expects further integration in KServe 0.12 with proposed <a class="" href="https://github.com/kserve/open-inference-protocol/pull/7" rel="noopener noreferrer" target="_blank">generate endpoint</a> for open inference protocol.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="-modelmesh-updates">📊 ModelMesh Updates<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-modelmesh-updates" title="Direct link to 📊 ModelMesh Updates">​</a></h2>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-storing-models-on-kubernetes-persistent-volumes-pvc">💾 Storing Models on Kubernetes Persistent Volumes (PVC)<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-storing-models-on-kubernetes-persistent-volumes-pvc" title="Direct link to 💾 Storing Models on Kubernetes Persistent Volumes (PVC)">​</a></h3>
<p>ModelMesh now allows to <a class="" href="https://github.com/kserve/modelmesh-serving/blob/main/docs/predictors/setup-storage.md#deploy-a-model-stored-on-a-persistent-volume-claim" rel="noopener noreferrer" target="_blank">directly mount model files onto serving runtimes pods</a>
using <a class="" href="https://kubernetes.io/docs/concepts/storage/persistent-volumes/" rel="noopener noreferrer" target="_blank">Kubernetes Persistent Volumes</a>. Depending on the selected <a class="" href="https://kubernetes.io/docs/concepts/storage/storage-classes/" rel="noopener noreferrer" target="_blank">storage solution</a> this approach can significantly reduce latency when deploying new predictors,
potentially remove the need for additional S3 cloud object storage like AWS S3, GCS, or Azure Blob Storage altogether.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-horizontal-pod-autoscaling-hpa">⚡ Horizontal Pod Autoscaling (HPA)<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-horizontal-pod-autoscaling-hpa" title="Direct link to ⚡ Horizontal Pod Autoscaling (HPA)">​</a></h3>
<p>Kubernetes <a class="" href="https://kubernetes.io/docs/tasks/run-application/horizontal-pod-autoscale/" rel="noopener noreferrer" target="_blank">Horizontal Pod Autoscaling</a> can now be used at the serving runtime pod level. With HPA enabled, the ModelMesh controller no longer manages the number of replicas. Instead, a <code>HorizontalPodAutoscaler</code> automatically updates the serving
runtime deployment with the number of Pods to best match the demand.</p>
<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-model-metrics-metrics-dashboard-payload-event-logging">📈 Model Metrics, Metrics Dashboard, Payload Event Logging<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#-model-metrics-metrics-dashboard-payload-event-logging" title="Direct link to 📈 Model Metrics, Metrics Dashboard, Payload Event Logging">​</a></h3>
<p>ModelMesh v0.11 introduces a new configuration option to emit a subset of useful metrics at the individual model level. These metrics can help identify outlier or "heavy hitter" models and consequently fine-tune the deployments of those inference services, like allocating more resources or increasing the number of replicas for improved responsiveness or avoid frequent cache misses.</p>
<p>A new <a class="" href="https://github.com/kserve/modelmesh-serving/blob/main/docs/monitoring.md#import-the-grafana-dashboard" rel="noopener noreferrer" target="_blank">Grafana dashboard</a> was added to display the comprehensive set of <a class="" href="https://github.com/kserve/modelmesh-serving/blob/main/docs/monitoring.md" rel="noopener noreferrer" target="_blank">Prometheus metrics</a> like model loading
and unloading rates, internal queuing delays, capacity and usage, cache state, etc. to monitor the general health of the ModelMesh Serving deployment.</p>
<p>The new <a class="" href="https://github.com/kserve/modelmesh/blob/main/src/main/java/com/ibm/watson/modelmesh/payload/" rel="noopener noreferrer" target="_blank"><code>PayloadProcessor</code> interface</a> can be implemented to log prediction requests and responses, to create data sinks for data visualization, for model quality assessment, or for drift and outlier detection by external monitoring systems.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="️-whats-changed">⚠️ What's Changed?<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.11-release#%EF%B8%8F-whats-changed" title="Direct link to ⚠️ What's Changed?">​</a></h2>
<ul>
<li class="">
<p>To allow longer InferenceService name due to DNS max length limits from <a class="" href="https://github.com/kserve/kserve/issues/1397" rel="noopener noreferrer" target="_blank">issue</a>, the <code>Default</code> suffix in the inference service component(predictor/transformer/explainer) name has been removed for newly created InferenceServices.
This affects the client that is using the component url directly instead of the top level InferenceService url.</p>
</li>
<li class="">
<p>Status.address.url is now consistent for both serverless and raw deployment mode, the url path portion is dropped in serverless mode.</p>
</li>
<li class="">
<p>Raw bytes are now accepted in v1 protocol, setting the right content-type header to <code>application/json</code> is required to recognize and decode the json payload if <code>content-type</code> is specified.</p>
