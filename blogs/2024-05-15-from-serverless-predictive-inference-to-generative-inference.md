<p><em>Published on May 15, 2024</em></p>
<p>We are excited to unveil KServe v0.13, marking a significant leap forward in evolving cloud native model serving to meet the demands of Generative AI inference. This release is highlighted by three pivotal updates: enhanced Hugging Face runtime, robust vLLM backend support for Generative Models, and the integration of OpenAI protocol standards.</p>
<p><img alt="!kserve-components" class="img_ev3q" height="540" src="https://kserve.github.io/website/assets/images/kserve-layer-08feccc0300cf8608f0a36b6572e70fb.png" width="960" /></p>
<p>Below are a summary of the key changes.</p>
<h2 class="anchor anchorTargetStickyNavbar_Vzrq" id="-enhanced-hugging-face-runtime-support">🚀 Enhanced Hugging Face Runtime Support<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.13-release#-enhanced-hugging-face-runtime-support" title="Direct link to 🚀 Enhanced Hugging Face Runtime Support">​</a></h2>
<p>KServe v0.13 enriches its Hugging Face runtime and now supports running Hugging Face models out-of-the-box. KServe v0.13 implements a <a class="" href="https://github.com/kserve/kserve/tree/release-0.13/python/huggingfaceserver" rel="noopener noreferrer" target="_blank">KServe Hugging Face Serving Runtime</a>, <code>kserve-huggingfaceserver</code>. With this implementation, KServe can now automatically infer a <a class="" href="https://huggingface.co/tasks" rel="noopener noreferrer" target="_blank">task</a> from model architecture and select the optimized serving runtime. Currently supported tasks include sequence classification, token classification, fill mask, text generation, and text to text generation.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-vllm-support">🔧 vLLM Support<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.13-release#-vllm-support" title="Direct link to 🔧 vLLM Support">​</a></h3>
<p>Version 0.13 introduces dedicated runtime support for <a class="" href="https://docs.vllm.ai/en/latest/" rel="noopener noreferrer" target="_blank">vLLM</a>, for enhanced transformer model serving. This support now includes auto-mapping vLLMs as the backend for supported tasks, streamlining the deployment process and optimizing performance. If vLLM does not support a particular task, it will default to the Hugging Face backend. See example below.</p>

<h3 class="anchor anchorTargetStickyNavbar_Vzrq" id="-openai-schema-integration">🌐 OpenAI Schema Integration<a class="hash-link" href="https://kserve.github.io/website/blog/kserve-0.13-release#-openai-schema-integration" title="Direct link to 🌐 OpenAI Schema Integration">​</a></h3>
<p>Embracing the OpenAI protocol, KServe v0.13 now supports three specific endpoints for generative transformer models:</p>
<ul>
<li class=""><code>/openai/v1/completions</code></li>
<li class=""><code>/openai/v1/chat/completions</code></li>
<li class=""><code>/openai/v1/models</code></li>
</ul>
<p>These endpoints are useful for generative transformer models, which take in messages and return a model-generated message output. The <a class="" href="https://platform.openai.com/docs/guides/text-generation/chat-completions-api" rel="noopener noreferrer" target="_blank">chat completions endpoint</a> is designed for easily handling multi-turn conversations, while still being useful for single-turn tasks. The <a class="" href="https://platform.openai.com/docs/guides/text-generation/completions-api" rel="noopener noreferrer" target="_blank">completions endpoint</a> is now a legacy endpoint that differs with the chat completions endpoint in that the interface for completions is a freeform text string called a <code>prompt</code>. Read more about the <a class="" href="https://platform.openai.com/docs/api-reference/chat" rel="noopener noreferrer" target="_blank">chat completions</a> and <a class="" href="https://platform.openai.com/docs/api-reference/completions" rel="noopener noreferrer" target="_blank">completions</a> endpoints in the OpenAI API docs.</p>
<p>This update fosters a standardized approach to transformer model serving, ensuring compatibility with a broader spectrum of models and tools, and enhances the platform's versatility. The API can be directly used with OpenAI's client libraries or third-party tools, like LangChain or LlamaIndex.</p>
